# 개발기획서 보완안: 멀티 AI 제공자 API 키 설정 지원

## 1) 보완 목표
설정 화면에서 사용자가 `Claude / ChatGPT / Gemini / Grok` 중 원하는 제공자를 선택하고 API 키를 입력하면, 해당 제공자로 정보글 생성 기능이 동작하도록 확장한다.

## 2) 제품 요구사항(추가)
1. 설정에서 AI 제공자 단일 선택 또는 우선순위 선택 지원
2. 제공자별 API 키 입력/저장/수정/삭제 기능
3. API 키 유효성 검사(저장 전 테스트 호출)
4. 생성 요청 시 선택된 제공자로 라우팅
5. 실패 시 대체 제공자(Fallback) 사용 여부 옵션
6. 제공자별 모델 선택 옵션(예: 고속/고품질)
7. 제공자별 비용/토큰 사용량 표시(대략 추정 포함)

## 3) 설정 UX(추가)
1. `AI 제공자` 섹션
- 제공자 선택: Claude, ChatGPT, Gemini, Grok
- 모드: `단일 제공자` / `우선순위 자동전환`

2. `API 키 관리` 섹션
- 키 입력 필드(마스킹/보기 토글)
- `연결 테스트` 버튼
- 마지막 검증 시각, 상태 배지(성공/실패)

3. `생성 정책` 섹션
- 실패 시 자동전환 ON/OFF
- 최대 재시도 횟수
- 모델/온도/출력 길이 옵션

## 4) 기술 설계(추가)
1. `LLM Adapter` 계층 도입
- 공통 인터페이스: `generateThread(input, options)`
- Provider Adapter: `AnthropicAdapter`, `OpenAIAdapter`, `GeminiAdapter`, `GrokAdapter`

2. `Routing Engine`
- 사용자 설정 기반 제공자 선택
- 실패 코드(429/5xx/timeout) 시 정책 기반 전환

3. `Prompt Normalizer`
- 제공자별 파라미터 차이 흡수
- 출력 포맷(JSON schema) 강제

## 5) 보안/키 관리 정책(중요)
1. 원칙: API 키는 클라이언트 평문 저장 금지
2. 권장 구조: 확장앱 → 백엔드 프록시 → 각 LLM API
3. 저장 시 암호화(서버 KMS/비밀관리 시스템)
4. 전송 시 TLS, 로그 마스킹, 키 원문 로그 금지
5. 사용자 삭제 요청 시 키 즉시 파기
6. 최소 권한 및 감사 로그 유지

## 6) 데이터 모델(추가)
- `user_ai_settings`
`user_id`, `primary_provider`, `fallback_enabled`, `provider_priority`, `updated_at`
- `user_api_keys`
`user_id`, `provider`, `encrypted_key`, `key_status`, `last_validated_at`
- `generation_logs`
`user_id`, `provider_used`, `model`, `latency_ms`, `token_in`, `token_out`, `result_status`

## 7) 운영 정책(추가)
1. 제공자별 Rate Limit 감지 및 백오프
2. 장애 시 상태 페이지/알림
3. 월별 비용 상한선 옵션(사용자 보호)
4. 제공자별 정책 변경 대응 버전 관리

## 8) QA 항목(추가)
1. 제공자별 키 검증 성공/실패 테스트
2. 동일 입력에서 출력 스키마 일관성 테스트
3. 장애/타임아웃 시 Fallback 동작 테스트
4. 키 마스킹/삭제/재등록 보안 테스트
5. 다국어 페이지 입력 안정성 테스트

## 9) 일정 영향(기존 8주 기준 조정)
1. +1주: 멀티 Provider Adapter + 라우팅 엔진
2. +1주: 키 관리/보안/검증 UI
3. +0.5주: 통합 QA 및 장애 시나리오 테스트
총 9.5~10주 권장

- 제공자별 기본 모델 선택
